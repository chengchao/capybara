Trusted control inside the guest

supervisor.py (root, TCB)
The only process the host talks to in the VM. Manages session users, mounts, sandboxes. No LLM, no planning, no policy — just a narrow privileged API.
  • ping / create_session / connect_directory
  • run_as_session / delete_session
  • cleanup_orphans / shutdown
/var/lib/capybara/sessions/<sid>/
Per-session state on disk.
  • home/ → bwrap /home/capybara
  • work/ → bwrap /workspace
  • mounts.json (root:root 0600)
capybara_<sid> users
Per-session unprivileged Linux user. Persistent across launches. Defense-in-depth under bwrap's namespace boundary.
/host-home (virtiofs mount)
User's home dir, mounted into the VM. Connected sub-paths get re-bound into per-session sandbox at /mnt/<name>.
↓ sudo -u capybara_<sid> -- bwrap … · per-call sandbox
  Tier 5 — bubblewrap sandbox Per-call · ephemeral · die-with-parent

Where agent commands actually run

bwrap process tree
--new-session --die-with-parent --unshare-pid · fresh /proc, /dev, /tmp · ro-bind of /usr, /bin, /lib, /etc
  • /workspace ← session work dir
  • /home/capybara ← session home dir
  • /mnt/<name> ← connected host dirs (rw or ro)
  • $HOME=/home/capybara, USER=capybara, TMPDIR=/tmp
bash -lc <agent's command>
The actual command Claude wanted to run. Sees only the sandbox's paths. Cannot reach /host-home or other sessions. Network IS shared with host — agents need DNS / pip / apt.

Trust boundary

Tier 5 is the security perimeter. The agent's LLM-decided commands cannot escape bwrap's mount + PID namespace. Per-session Linux user is defense-in-depth under that boundary, not the primary gate.

One supervisor channel

Main process owns a single persistent SupervisorClient. Both control-plane calls (create/connect/delete) and agent tool calls (run_as_session) share it. Promise-chain serializes; the supervisor is itself strictly serial.

Tool interception

The SDK ships Bash/Read/Glob as built-in tools that would run on the host. Our in-process MCP server shadows them by name (prefixed mcp__capybara__*); the SDK's allowedTools allow-list keeps only ours, blocking the unsafe defaults.

Three Bun roles

(1) Bun is the dev-tooling runtime: bun install, bun build. (2) Electron's main runs on Node 22, NOT Bun. (3) A bundled Bun binary is shipped to satisfy the SDK's executable: "bun" for the spawned cli.js.

Lifecycle of a single agent task

  1. User types a prompt in AgentProbe and hits Run.
  2. Renderer calls window.capybara.startAgentTask(prompt), which crosses contextBridge → preload → ipcRenderer.invoke("start-agent-task", prompt).
  3. Main process assigns a taskId, attaches an AbortController, calls runAgentTask(prompt, taskId, emit, controller), returns the taskId.
  4. runAgentTask ensures the session exists (idempotent create_session via SupervisorClient), then calls query() with the MCP server + allowed tools list.
  5. SDK spawns the claude binary as a subprocess (under bundled Bun). Sends prompt + tool defs.
  6. Claude emits an assistant message. If it contains a tool_use block (e.g. Bash), the SDK calls back into main's in-process MCP handler.
  7. Tool handler calls supervisor.request("run_as_session", { command, … }) via the persistent SupervisorClient.
  8. Supervisor spawns sudo -u capybara_<sid> -- bwrap … bash -lc <command> inside the VM.
  9. Command runs in the sandbox, exits. supervisor.py collects stdout/stderr/exitCode and returns the JSON-RPC response.
  10. Tool result flows back to the SDK → Claude continues reasoning → repeat until done.
  11. Each event (assistant_message, tool_use, tool_result) hits webContents.send("agent-event", payload) and surfaces in the renderer.
